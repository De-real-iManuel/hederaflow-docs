## Core User Flows

### 1. User Registration & Wallet Connection

```mermaid
sequenceDiagram
    participant U as User
    participant F as Frontend
    participant B as Backend
    participant H as Hedera
    participant DB as Database
    
    U->>F: Enter email & password
    F->>B: POST /auth/register
    B->>DB: Create user record
    B->>H: Create Hedera account
    H-->>B: Account ID + Private Key
    B->>DB: Store Hedera account ID
    B-->>F: JWT token + user data
    F->>F: Store token in localStorage
    
    Note over U,F: Optional: HashPack Wallet Connection
    U->>F: Click "Connect HashPack"
    F->>HashPack: Request connection
    HashPack-->>F: Account ID + signature
    F->>B: POST /auth/wallet-connect
    B->>H: Verify signature
    H-->>B: Signature valid
    B->>DB: Link wallet to user
    B-->>F: Updated user data
```

### 2. Meter Registration

```mermaid
sequenceDiagram
    participant U as User

    U->>F: Enter meter details
```

### 3. Meter Reading Verification (Core Flow)

```mermaid
sequenceDiagram
    participant U as User
    participant F as Frontend
    participant B as Backend
    participant OCR as Google Vision
    participant FD as Fraud Detection
    participant IPFS as Pinata
    participant HCS as Hedera HCS
    participant DB as Database
    
    U->>F: Capture meter photo
    F->>F: Client-side OCR (Tesseract)
    
    alt Client OCR confidence >= 90%
        F->>B: POST /verify (with OCR result)
    else Client OCR failed
        F->>B: POST /verify (image only)
        B->>OCR: Extract text
        OCR-->>B: Reading + confidence
    end
    
    B->>DB: Get previous reading
    DB-->>B: Previous reading value
    B->>B: Calculate consumption
    
    B->>FD: Analyze image + reading
    FD->>FD: ELA analysis
    FD->>FD: Anomaly detection
    FD-->>B: Fraud score + flags
    
    B->>IPFS: Upload image
    IPFS-->>B: IPFS hash
    
    B->>HCS: Log verification
    Note over HCS: Type: VERIFICATION<br/>Reading, Confidence,<br/>Fraud Score, Status
    HCS-->>B: Sequence number
    
    B->>DB: Save verification
    
    alt Consumption > 0
        B->>B: Calculate bill
        B->>DB: Create bill record
    end
    
    B-->>F: Verification result
    F->>U: Show result + bill
```

### 4. Bill Payment Flow

```mermaid
sequenceDiagram
    participant U as User
    participant F as Frontend
    participant B as Backend
    participant R as Redis
    participant CG as CoinGecko
    participant HP as HashPack
    participant H as Hedera
    participant MN as Mirror Node
    participant HCS as HCS Topic
    participant DB as Database
    
    U->>F: Click "Pay Bill"
    F->>B: POST /payments/prepare
    B->>CG: Get HBAR price
    CG-->>B: Price in fiat
    B->>B: Calculate HBAR amount<br/>(+2% buffer)
    B->>R: Create rate lock (5 min)
    R-->>B: Lock created
    B-->>F: Transaction details
    
    F->>U: Show payment details
    U->>F: Confirm payment
    F->>HP: Request transaction signature
    HP->>U: Approve in wallet
    U->>HP: Approve
    HP->>H: Submit HBAR transfer
    H-->>HP: Transaction ID
    HP-->>F: Transaction ID
    
    F->>B: POST /payments/confirm
    B->>R: Validate rate lock
    R-->>B: Lock valid
    B->>MN: Verify transaction
    MN-->>B: Transaction confirmed
    B->>B: Validate amount
    
    B->>HCS: Log payment
    Note over HCS: Type: PAYMENT<br/>Bill ID, Amount HBAR,<br/>Amount Fiat, Exchange Rate
    HCS-->>B: Sequence number
    
    B->>DB: Update bill status = PAID
    B->>R: Delete rate lock
    B-->>F: Payment receipt
    F->>U: Show success + receipt
```

## Data Flow Patterns

### 1. Caching Strategy

```mermaid
graph LR
    A[API Request] --> B{Cache Hit?}
    B -->|Yes| C[Return Cached Data]
    B -->|No| D[Fetch from Source]
    D --> E[Store in Redis]
    E --> F[Return Fresh Data]
    
    style B fill:#ffd
    style E fill:#9cf
```

**Cache TTLs:**
- Exchange rates: 5 minutes
- Tariff data: 1 hour
- User preferences: 15 minutes
- Rate locks: 5 minutes (exact)

### 2. Error Handling Flow

```mermaid
graph TB
    A[API Request] --> B{Validation}
    B -->|Invalid| C[400 Bad Request]
    B -->|Valid| D{Authentication}
    D -->|Failed| E[401 Unauthorized]
    D -->|Success| F{Authorization}
    F -->|Denied| G[403 Forbidden]
    F -->|Allowed| H{Business Logic}
    H -->|Error| I[500 Internal Error]
    H -->|Success| J[200 OK]
    
    I --> K[Log Error]
    K --> L[Rollback Transaction]
    
    style C fill:#f99
    style E fill:#f99
    style G fill:#f99
    style I fill:#f99
    style J fill:#9f9
```

### 3. Transaction Verification

```mermaid
graph TB
    A[Transaction ID] --> B[Mirror Node Query]
    B --> C{Transaction Found?}
    C -->|No| D[404 Not Found]
    C -->|Yes| E{Status = SUCCESS?}
    E -->|No| F[400 Transaction Failed]
    E -->|Yes| G{Amount Match?}
    G -->|No| H[400 Amount Mismatch]
    G -->|Yes| I{Recipient Match?}
    I -->|No| J[400 Invalid Recipient]
```
